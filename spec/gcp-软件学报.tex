\begin{center}
  \textbf{Abstract}
\end{center}

\noindent
Type inference for dynamically-typed languages faces a fundamental challenge: the type of a function parameter is simultaneously constrained by four independent sources---the values assigned to it in the function body, the programmer's explicit declaration, the argument types passed at call sites, and the structural operations (field access, method calls) performed on it inside the body. Existing approaches, from Hindley-Milner unification to TypeScript's bidirectional checking, conflate these sources into a single constraint set, producing spurious conflicts and forcing developers to supply redundant annotations.

\medskip\noindent
We present \textbf{Generalized Constraint Projection (GCP)}, a type inference methodology that models each source as an independent constraint dimension (\texttt{extendToBe}, \texttt{declaredToBe}, \texttt{hasToBe}, \texttt{definedToBe}) propagated separately via lattice operations and resolved jointly at projection time. We formalize GCP on the \textbf{Outline} language, defining a type lattice with OEM structural subtyping rules $\NOTHING \sqsubseteq \tau$ (nothing is everything) and $\tau \sqsubseteq \ANY$ (everything is any), a four-dimensional Genericable constraint model with monotonicity and convergence proofs, and a bidirectional projection mechanism for polymorphic higher-order function instantiation. We prove soundness, local and global convergence (fixed-point iteration terminates in $O(N \cdot |\mathbb{T}|)$ steps), and order-independence.

\medskip\noindent
We apply GCP to zero-annotation Python AOT compilation via \textbf{GCP-Python}. The \texttt{mypyc} compiler can compile type-annotated Python to C extensions, but its practical barrier is \emph{annotation coverage}: unannotated functions yield negligible gains (1.30$\times$ average; one case degrades to 0.81$\times$). GCP-Python is a demand-driven inference pipeline that automatically injects PEP~484 annotations in \textbf{66\,ms}. Evaluated across 22 Python program categories, GCP-Python achieves a \textbf{17.1$\times$} arithmetic-mean speedup over CPython, with peaks of \textbf{120$\times$} (match/case dispatch) and \textbf{34$\times$} on real-world number-theory code. On a 7-function oracle benchmark, GCP-Python reaches \textbf{101.8\%} of hand-annotated performance on ARM64 and \textbf{99.7\%} on Linux x86-64, and \textbf{outperforms warmed-up Numba JIT} (\texttt{is\_prime(997)}: 14.67$\times$ vs.\ 7.66$\times$).

\medskip\noindent
\textbf{Key words:} type inference; dynamic languages; constraint solving; ahead-of-time compilation; Python; structural type systems

\begin{center}
  \textbf{摘\ 要}
\end{center}

\noindent
动态编程语言的类型推导面临一个根本困难：函数参数的类型信息来自四个性质不同、相互独立的来源——函数体内的赋值决定了参数能承载什么值，程序员的显式声明给出了约束，调用点传入的实参类型反映了外部对参数的期望，而函数体内的结构化访问（字段读取、方法调用）则揭示了参数必须具备的最小结构。现有方法——从Hindley-Milner统一到TypeScript的双向检查——将这四类信息混入同一个约束集，导致虚假冲突并迫使开发者提供冗余标注。

本文提出\textbf{泛化约束投影（Generalized Constraint Projection，GCP）}：将上述四类信息分别建模为独立的约束维度（\texttt{extendToBe}、\texttt{declaredToBe}、\texttt{hasToBe}、\texttt{definedToBe}），通过格操作独立传播，在投影时联合求解。我们在\textbf{Outline}语言上形式化该方法，定义类型格（含OEM结构化子类型规则 $\NOTHING \sqsubseteq \tau$ 与 $\tau \sqsubseteq \ANY$）、四维Genericable约束模型及双向投影机制，并证明了健全性、局部与全局收敛性（不动点算法在 $O(N \cdot |\mathbb{T}|)$ 步内终止）以及顺序无关性。

我们将GCP应用于Python零标注AOT编译（\textbf{GCP-Python}）：在66毫秒内自动为零标注Python源文件注入PEP~484标注。在22个Python程序类别上，GCP-Python相对CPython实现算术平均\textbf{17.1倍}加速，峰值达120倍。在7函数oracle基准上，ARM64达手工标注性能的\textbf{101.8\%}，x86-64达\textbf{99.7\%}，且无需开发者工作；在调用密集型整数函数上\textbf{优于预热Numba JIT}（\texttt{is\_prime(997)}：14.67倍 vs 7.66倍）。

\medskip\noindent
\textbf{关键词：}类型推导；动态语言；约束求解；提前编译；Python；结构化类型系统

\subsection{研究背景与动机}

开发者生产力与类型安全之间的张力是编程语言设计中最古老的问题之一。动态语言倾向于生产力——变量无需类型标注，对象可以在运行时获得新属性，函数可以接受任何形状的参数。代价在运行时支付：本可在编译时捕获的类型错误反而在生产环境中表现为隐晦的失败。

渐进类型系统试图弥合这一鸿沟。TypeScript是占主导地位的工业实例，但仍表现出三个痛点：
\begin{enumerate}[leftmargin=2em]
\item \textbf{冗余标注}：即使表达式的类型从上下文中明确无误，TypeScript仍经常需要显式标注（例如高阶函数调用上的泛型类型参数）。
\item \textbf{上下文不敏感}：变量的类型在声明点固定，无法在两个独立的调用上下文中以不同方式收窄。
\item \textbf{繁琐的泛型推导}：编写正确的泛型高阶函数需要冗长、易错的参数级泛型声明。
\end{enumerate}

Python面临类似但更严重的挑战。\texttt{mypyc}编译器为弥合性能差距提供了一条引人注目的路径：给定带有PEP~484类型标注的Python源代码，\texttt{mypyc}将每个标注的函数编译为C扩展（\texttt{.so}），消除装箱开销并用直接的C函数调用替换动态方法分派。问题在于\emph{标注覆盖率}：当\texttt{mypyc}遇到未标注的参数时，它生成保守的装箱代码，几乎不优于解释器。

GCP的核心洞察是：四个约束方向——\emph{实际赋值、程序员标注、调用点需求、结构化访问模式}——提供互补信息，它们的联合最小化唯一确定大多数类型而无需任何标注。

\subsection{现有方法的局限性}

\begin{itemize}[leftmargin=2em]
\item \textbf{mypy/Pyright}：声明点推导，无调用点信息无法推导参数类型。
\item \textbf{Cython}：需要带有显式C类型\texttt{cdef}声明的\texttt{.pyx}文件，与标准Python不兼容。
\item \textbf{Numba \texttt{@jit}}：通过LLVM跟踪在运行时特化，无法生成静态\texttt{.so}文件，首次调用产生JIT开销。
\item \textbf{TypeScript}：使用结构化类型系统，但未形式化完整的四维约束模型，高阶函数泛型推导通常需要显式标注。
\end{itemize}

GCP-Python占据了这些方法之间的空白：静态分析（无运行时开销）、调用点驱动（无手工标注）、标准兼容输出（PEP~484标注$\to$标准mypyc）。

\subsection{本文贡献}

\paragraph{理论贡献：}
\begin{enumerate}[leftmargin=2em]
\item \textbf{四维约束模型}（\S\ref{sec:model}）：将四种性质不同的类型信息源分离为独立的约束维度并独立传播，避免虚假约束冲突。
\item \textbf{OEM结构化子类型}（\S\ref{sec:oem}）：鸭子类型的形式化，含 $\NOTHING \sqsubseteq \tau$（nothing is everything）和 $\tau \sqsubseteq \ANY$（everything is any）两条基础规则。双向委托协议（\texttt{tryIamYou}/\texttt{tryYouAreMe}）使OEM成为开放扩展点。
\item \textbf{双向投影}（\S\ref{sec:projection}）：泛型实例化机制，同时从调用点向lambda参数传播类型信息并从lambda体向返回类型传播。
\item \textbf{形式化保证}：健全性（定理~\ref{thm:soundness}）、局部与全局收敛性（定理~\ref{thm:convergence}）、顺序无关性（定理~\ref{thm:order}）。
\end{enumerate}

\section{相关工作}
\label{sec:related}
%% ============================================================

\subsection{类型推导理论}

基础工作是Milner的算法W~\cite{milner1978}及其对递归类型的扩展~\cite{damas1982}。GCP的基于约束的方法与Pottier和Rémy~\cite{pottier2005}的工作更密切相关，他们将HM推广到支持子类型的基于约束的框架。

经典的基于约束的类型推导生成一组等式或子类型约束并同时求解。根本困难在于，动态语言中存在\emph{四种性质不同的类型信息源}，它们在推导过程遍历AST时异步到达。将它们视为单一的无差别约束集混淆了它们的不同角色，导致虚假的约束冲突。

Hindley-Milner(HM)类型推导算法通过Robinson统一生成全局类型等式约束集并求解。GCP在三个基本方面有所不同：

\begin{table}[h]
\centering
\caption{GCP与Hindley-Milner的比较}
\label{tab:hm-vs-gcp}
\begin{tabular}{lll}
\toprule
\textbf{维度} & \textbf{Hindley-Milner} & \textbf{GCP} \\
\midrule
约束结构 & 类型等式集 $\tau = \sigma$ & 每个函数参数四个方向不等式 \\
求解方法 & 全局统一（Robinson） & 局部每参数约束收窄 \\
增量更新 & 变化时需完全重新推导 & 单变量约束添加是$O(1)$ \\
动态扩展 & 无原生支持 & \texttt{extendToBe}捕获声明后赋值 \\
渐进类型 & 通过显式\texttt{dyn}类型 & 通过\texttt{NOTHING}/\texttt{ANY}初始值 \\
\bottomrule
\end{tabular}
\end{table}

\subsection{渐进类型系统}

Pierce和Turner~\cite{pierce2000}的渐进类型系统及其后续阐述~\cite{siek2006}允许混合类型化和非类型化代码。GCP的\texttt{NOTHING}类型和惰性推导在渐进系统中扮演与\texttt{?}类型类似的角色，但完全是内部的——它们不出现在面向用户的类型语言中。

\subsection{结构化类型系统}

GCP的OEM是鸭子类型的形式化，与类型理论中的记录子类型~\cite{pierce2002}密切相关，但扩展到处理循环结构化类型。与GCP最可比的工业规模类型系统是TypeScript~\cite{typescript2012}。理解TypeScript不健全性的工作~\cite{bierman2014}促使了GCP更有纪律的约束模型。Facebook的Flow~\cite{flow2017}对JavaScript应用类似的结构化类型系统，但与TypeScript一样，它没有分离GCP维护的四个类型信息维度。

\subsection{需求驱动类型推导}

从使用点向定义点传播类型需求与Mycroft~\cite{mycroft1984}的多态类型推导以及基于约束推导的\emph{自底向上}变体密切相关。GCP的\texttt{hasToBe}传播是这一传统中的调用点需求约束，应用于动态类型宿主语言。

\subsection{Python编译技术}

\textbf{Cython}~\cite{cython2007}需要手工C类型声明。\textbf{Numba}~\cite{numba2015}使用LLVM JIT在调用时特化，产生JIT开销和非确定性编译产物。\textbf{Nuitka}~\cite{nuitka2012}是一个提前Python到C++编译器，不需要标注但执行全程序分析，不与mypyc \texttt{.so}模型集成。\textbf{mypy}~\cite{mypy2012}的推导是\emph{声明点驱动}的，忽略调用点实际传递的类型。\textbf{PyPy}~\cite{pypy2009}通过元跟踪JIT实现5--10倍平均加速，但作为替代运行时，无法生成与CPython生态系统兼容的标准\texttt{.so} C扩展。

\section{GCP方法论}
\label{sec:gcp}
%% ============================================================

\subsection{动机与核心思想}
\label{sec:motivation}

动态语言的类型推导面临一个根本挑战：同一个函数参数的类型信息来自四个性质不同的源：

\begin{enumerate}[leftmargin=2em]
\item \textbf{值源}：参数在函数体内被赋予什么值？（\texttt{x = 42} $\Rightarrow$ \texttt{x}的\texttt{extendToBe}至少是\texttt{Int}）
\item \textbf{声明源}：程序员声明了什么类型？（\texttt{x: Number} $\Rightarrow$ \texttt{x}的\texttt{declaredToBe}是\texttt{Number}）
\item \textbf{需求源}：调用点传递什么类型的实参？（\texttt{f(10)} $\Rightarrow$ \texttt{f}的参数\texttt{x}的\texttt{hasToBe}收集到\texttt{Int}）
\item \textbf{结构源}：参数如何被访问？（\texttt{x.name} $\Rightarrow$ \texttt{x}的\texttt{definedToBe}必须有\texttt{name}字段）
\end{enumerate}

GCP的核心思想：\textbf{将四个信息源分离为独立的约束维度，独立传播，最后通过格操作联合求解}。

\subsection{四维约束模型}
\label{sec:model}

GCP中的每个函数参数都表示为一个\textbf{Genericable}对象，携带四个约束槽。

\begin{definition}[Genericable约束元组]
对于函数参数$x$，其约束元组为 $C(x) = (\tau_e, \tau_d, \tau_h, \tau_f)$，其中各维度的初始值、更新操作和收敛方向如表~\ref{tab:constraints}所示。
\end{definition}

\begin{table}[h]
\centering
\caption{四维约束槽定义}
\label{tab:constraints}
\begin{tabular}{lllll}
\toprule
\textbf{槽位} & \textbf{符号} & \textbf{初始值} & \textbf{更新操作} & \textbf{收敛方向} \\
\midrule
\texttt{extendToBe}  & $\tau_e$ & $\bot$ (\texttt{NOTHING}) & $\tau_e := \tau_e \sqcup \tau$ & 从$\bot$向上收敛 \\
\texttt{declaredToBe} & $\tau_d$ & $\top$ (\texttt{ANY})     & $\tau_d := \tau$（直接赋值）   & 一次性设置 \\
\texttt{hasToBe}     & $\tau_h$ & $\top$ (\texttt{ANY})     & $\tau_h := \tau_h \sqcap \tau$ & 从$\top$向下收敛 \\
\texttt{definedToBe} & $\tau_f$ & $\top$ (\texttt{ANY})     & $\tau_f := \tau_f \sqcap \tau$ & 从$\top$向下收敛 \\
\bottomrule
\end{tabular}
\end{table}

\noindent 约束添加操作的形式化定义：

\begin{align}
\texttt{addExtend}(\tau)&:\ \tau_e := \tau_e \sqcup \tau \quad \text{// join：从$\bot$向上扩展，收集所有赋值类型的最小上界} \label{eq:addextend}\\
\texttt{addDeclared}(\tau)&:\ \tau_d := \tau \quad \text{// 直接设置} \label{eq:adddeclared}\\
\texttt{addDemand}(\tau)&:\ \tau_h := \tau_h \sqcap \tau \quad \text{// meet：从$\top$向下收紧，调用点需求的交集} \label{eq:adddemand}\\
\texttt{addStructure}(\tau)&:\ \tau_f := \tau_f \sqcap \tau \quad \text{// meet：从$\top$向下收紧，结构访问需求的交集} \label{eq:addstructure}
\end{align}

\paragraph{$\hasToBe$与$\definedToBe$的合并。}
两者是\textbf{并行的上界约束}，分别来自调用点需求和函数体结构访问，投影时必须同时满足。合并为 $\tau_h \sqcap \tau_f$（meet），语义是字段取\textbf{并集}——结果比两个操作数都更严格（子类型）：

\[
\tau_h \sqcap \tau_f = \begin{cases}
\tau_f & \text{若 } \tau_h = \ANY \\
\tau_h & \text{若 } \tau_f = \ANY \\
\tau_h & \text{若 } \tau_h \sqsubseteq \tau_f \text{（取更严格者）} \\
\tau_f & \text{若 } \tau_f \sqsubseteq \tau_h \text{（取更严格者）} \\
\text{结构合并（字段取并集）} & \text{若两者均为ProductADT} \\
\text{类型错误} & \text{其他（类型根本不兼容）}
\end{cases}
\]

\paragraph{完整约束链。}
函数定义时的合法性条件（从最严格到最宽松，左边是子类型，右边是父类型）：
\begin{equation}
\tau_e \sqsubseteq \tau_d \sqsubseteq \tau_h \sqcap \tau_f
\label{eq:chain-def}
\end{equation}

函数调用时的投影条件（实参类型$\tau_{\mathit{arg}}$必须落在约束链中）：
\begin{equation}
\tau_e \sqsubseteq \tau_{\mathit{arg}} \sqsubseteq \tau_d \sqsubseteq \tau_h \sqcap \tau_f
\label{eq:chain-call}
\end{equation}

语义：
\begin{itemize}[leftmargin=2em]
\item $\tau_e \sqsubseteq \tau_{\mathit{arg}}$：实参必须是\texttt{extendToBe}的父类型，覆盖函数体内所有赋值类型
\item $\tau_{\mathit{arg}} \sqsubseteq \tau_d$：实参必须满足程序员的显式声明（实参是声明的子类型）
\item $\tau_d \sqsubseteq \tau_h \sqcap \tau_f$：声明必须满足调用点需求与结构访问需求的交集
\end{itemize}

若$\tau_{\mathit{arg}}$不满足此链，投影失败并报告类型错误。

\paragraph{案例研究：函数参数的四维约束收敛过程。}

考虑以下Outline函数：

\begin{lstlisting}[language={}]
let processUser = user -> {
    user = {name: "Alice", age: 25, email: "alice@example.com"};
    let userName = user.name;
    let userAge  = user.age;
    sendEmail(user);   // sendEmail 需要 {email: String}
    return user.name;
}
processUser({name: "Bob", age: 30, email: "bob@example.com"});
\end{lstlisting}

图~\ref{fig:convergence}展示了参数\texttt{user}的四维约束收敛过程。

\begin{figure}[h]
\centering

\subsection{单调性与收敛性}
\label{sec:convergence}

\begin{theorem}[单调性]
\label{thm:monotone}
每次约束添加操作都是单调的：
\begin{itemize}[leftmargin=2em]
\item \texttt{addExtend}通过join($\sqcup$)单调递增：$\tau_e$从$\bot$向上收敛（收集所有赋值类型的最小上界）
\item \texttt{addDemand}和\texttt{addStructure}通过meet($\sqcap$)单调递减：$\tau_h$、$\tau_f$从$\top$向下收敛（需求交集越来越严格）
\item \texttt{declaredToBe}只设置一次，不参与迭代
\end{itemize}
\end{theorem}

\begin{theorem}[局部收敛性]
\label{thm:local-conv}
对于单个参数，约束序列在有限步内收敛：
\begin{itemize}[leftmargin=2em]
\item $\tau_e$从$\bot$出发，每次join操作单调递增，最多上升到$\top$，需$O(|\mathbb{T}|)$步
\item $\tau_h$和$\tau_f$从$\top$出发，每次meet操作单调递减，最多下降到$\bot$，需$O(|\mathbb{T}|)$步
\end{itemize}
在有限高度的类型格中，单个参数的推导在$O(|\mathbb{T}|)$步内终止。
\end{theorem}

\begin{theorem}[全局收敛性]
\label{thm:convergence}
对于$N$个函数参数的程序，推导算法在$O(N \cdot |\mathbb{T}|)$步内终止。
\end{theorem}

\begin{proof}
每个参数最多经历$|\mathbb{T}|$次约束更新，$N$个参数总共$O(N \cdot |\mathbb{T}|)$次更新。由于每次更新都是单调的且格有限高度，算法必然终止。\qed
\end{proof}

\begin{theorem}[健全性]
\label{thm:soundness}
如果$\projected(x) = \tau$，则在运行时$x$的值类型是$\tau$的子类型。
\end{theorem}

\begin{proof}
$\projected(x)$满足约束链$\tau_e \sqsubseteq \projected(x) \sqsubseteq \tau_d \sqsubseteq \tau_h \sqcap \tau_f$，因此：
\begin{itemize}[leftmargin=2em]
\item $\tau_e \sqsubseteq \projected(x)$：projected是extendToBe的父类型，覆盖函数体内所有赋值类型
\item $\projected(x) \sqsubseteq \tau_d$：projected满足程序员的显式声明
\item $\projected(x) \sqsubseteq \tau_h \sqcap \tau_f$：projected同时满足调用点需求和结构访问需求的交集
\end{itemize}
因此，$\projected(x)$是运行时类型的安全近似。\qed
\end{proof}

\subsection{Outline语言类型系统}
\label{sec:typesystem}

为了形式化GCP，我们定义\textbf{Outline}语言的类型系统。

\paragraph{类型语法：}
\[
\tau ::= \top \mid \bot \mid \mathit{prim} \mid \texttt{Entity}(n, [m_i:\tau_i]) \mid \texttt{Tuple}([f_i:\tau_i]) \mid \texttt{Array}(\tau) \mid \texttt{Dict}(\tau_k, \tau_v) \mid \texttt{Option}(\tau_1,\ldots,\tau_n) \mid \tau_1 \to \tau_2 \mid G[C] \mid \bullet \mid {?}
\]

其中$\mathit{prim} \in \{\texttt{String}, \texttt{Int}, \texttt{Long}, \texttt{Float}, \texttt{Double}, \texttt{Number}, \texttt{Bool}, \texttt{Symbol}\}$，且\texttt{Number}是\texttt{Int}$|$\texttt{Long}$|$\texttt{Float}$|$\texttt{Double}的声明超类型。

\paragraph{类型格：}完整的类型格$(\mathbb{T}, \sqsubseteq)$由结构化子类型关系定义：
\begin{itemize}[leftmargin=2em]
\item $\top$（\texttt{ANY}）是最大元素：$\forall\tau.\ \tau \sqsubseteq \top$
\item $\bot$（\texttt{NOTHING}）是最小元素：$\forall\tau.\ \bot \sqsubseteq \tau$
\item 数值提升：$\texttt{Int} \sqsubseteq \texttt{Number}$，$\texttt{Long} \sqsubseteq \texttt{Number}$，$\texttt{Float} \sqsubseteq \texttt{Number}$，$\texttt{Double} \sqsubseteq \texttt{Number}$
\item 实体继承：当B扩展A时，$\texttt{Entity}(B,\ldots) \sqsubseteq \texttt{Entity}(A,\ldots)$
\item 函数逆变：当$\tau_1 \sqsubseteq \sigma_1$且$\sigma_2 \sqsubseteq \tau_2$时，$(\sigma_1 \to \sigma_2) \sqsubseteq (\tau_1 \to \tau_2)$
\end{itemize}

\subsection{Outline表达式匹配（OEM）：结构化子类型}
\label{sec:oem}

OEM是GCP的结构化子类型关系，形式化了鸭子类型："如果它走起来像鸭子，叫起来像鸭子，那它就是鸭子。"

\begin{definition}[OEM关系]
类型$\tau$匹配类型$\sigma$（记作$\tau \sqsubseteq \sigma$）当且仅当$\tau$的实例可以安全地用在期望$\sigma$的上下文中。
\end{definition}

\paragraph{OEM规则：}
\begin{enumerate}[leftmargin=2em]
\item \textbf{自反性}：$\tau \sqsubseteq \tau$
\item \textbf{传递性}：若$\tau \sqsubseteq \sigma$且$\sigma \sqsubseteq \rho$，则$\tau \sqsubseteq \rho$
\item \textbf{顶底规则}（nothing is everything；everything is any）：
  \begin{itemize}
  \item $\forall\tau.\ \NOTHING \sqsubseteq \tau$\quad — NOTHING是所有类型的子类型（最严格，可赋给任意变量）
  \item $\forall\tau.\ \tau \sqsubseteq \ANY$\quad — 任何类型都是ANY的子类型（最宽松，接受任意值）
  \item 等价地：$\bot = \NOTHING$（格的底元素），$\top = \ANY$（格的顶元素）
  \end{itemize}
\item \textbf{原始类型提升}：$\texttt{Int} \sqsubseteq \texttt{Number}$，$\texttt{Long} \sqsubseteq \texttt{Number}$，$\texttt{Float} \sqsubseteq \texttt{Number}$，$\texttt{Double} \sqsubseteq \texttt{Number}$
\item \textbf{结构化记录子类型}（宽度子类型）：若$\forall i.\ \tau_i \sqsubseteq \sigma_i$，则
  \[\texttt{Tuple}([f_1:\tau_1,\ldots,f_n:\tau_n,\ldots]) \sqsubseteq \texttt{Tuple}([f_1:\sigma_1,\ldots,f_n:\sigma_n])\]
  即：子类型可以有更多字段（宽度），且对应字段类型协变
\item \textbf{函数子类型}（参数逆变，返回值协变）：若$\sigma_1 \sqsubseteq \tau_1$且$\tau_2 \sqsubseteq \sigma_2$，则$(\tau_1 \to \tau_2) \sqsubseteq (\sigma_1 \to \sigma_2)$
\item \textbf{数组协变}：若$\tau \sqsubseteq \sigma$，则$\texttt{Array}(\tau) \sqsubseteq \texttt{Array}(\sigma)$
\item \textbf{和类型}：若$\forall i.\ \tau_i \sqsubseteq \sigma$，则$\texttt{Option}(\tau_1,\ldots,\tau_n) \sqsubseteq \sigma$
\end{enumerate}

\begin{theorem}[OEM是预序]
\label{thm:oem-preorder}
OEM关系$(\mathbb{T}, \sqsubseteq)$是预序，即满足自反性和传递性。
\end{theorem}

\paragraph{双向委托协议：}OEM的实现使用双向委托协议，使其成为开放扩展点：

\begin{lstlisting}[language=Java]
interface OEMCapable {
  boolean tryIamYou(OutlineType other);   // "我能当作你吗?"
  boolean tryYouAreMe(OutlineType other); // "你能当作我吗?"
}
\end{lstlisting}

当检查$\tau \sqsubseteq \sigma$时：(1)首先尝试$\tau.\texttt{tryIamYou}(\sigma)$；(2)如果失败，尝试$\sigma.\texttt{tryYouAreMe}(\tau)$；(3)两者都失败则$\tau \sqsubseteq \sigma$不成立。这种双向协议允许新类型扩展OEM而无需修改现有类型的代码。

\subsection{泛型实例化：双向投影}
\label{sec:projection}

GCP的投影机制处理多态高阶函数的类型实例化。以下示例展示GCP如何通过双向投影检测深层结构化类型冲突：

\begin{lstlisting}[language={}]
// 高阶函数: 提升选择器和谓词的组合
let lift = sel -> pred -> entity -> pred(sel(entity))
let get_score  = player -> player.score
let is_passing = n -> n >= 60   // 需要 n.points : Int
let is_ace     = n -> n.score >= 90

let check_pass = lift(get_score)(is_passing)
let alice = {name="Alice", test={score=85, gpa=3.5}, rank=3}
let pass = check_pass(alice)
// 推导错误: alice.test 缺少 points 字段
\end{lstlisting}

GCP通过三层嵌套的双向投影自动完成：
\begin{enumerate}[leftmargin=2em]
\item \texttt{lift(get\_score)} $\Rightarrow$ 推导 \texttt{sel: \{score:Int\} $\to$ Int}
\item \texttt{lift(get\_score)(is\_passing)} $\Rightarrow$ 推导 \texttt{pred: Int $\to$ Bool}
\item \texttt{check\_pass(entity)} $\Rightarrow$ 反向传播 \texttt{entity} 必须有 \texttt{\{score: Int\}}
\end{enumerate}

\begin{theorem}[投影终止性]
\label{thm:projection}
投影算法在$O(|\mathit{params}| \cdot |\mathbb{T}|)$步内终止。
\end{theorem}

\subsection{类型推导算法}
\label{sec:algorithm}

GCP的类型推导算法专注于\textbf{函数参数的四维约束收集}和\textbf{调用点投影}。

\paragraph{推导对象：}
\begin{itemize}[leftmargin=2em]
\item[$\checkmark$] \textbf{函数参数}：创建Genericable，收集四维约束
\item[$\checkmark$] \textbf{函数返回值}：通过投影推导
\item[$\times$] \textbf{字面量}：直接有类型（如\texttt{42: Int}），不需要推导
\item[$\times$] \textbf{局部变量}：直接类型推导，不是Genericable
\end{itemize}

\paragraph{核心推导规则：}

\begin{enumerate}[leftmargin=2em]
\item \textbf{Lambda表达式}——创建参数的Genericable：
\begin{lstlisting}[language={}]
infer(x -> body):
  1. 创建 Genericable(x)
  2. 分析函数体,收集对x的约束:
     - 从赋值收集 extendToBe  (join⊔向上扩展)
     - 从调用点收集 hasToBe   (meet⊓向下收紧)
     - 从结构访问收集 definedToBe (meet⊓向下收紧)
  3. 检查约束: τ_e ⊑ τ_d ⊑ τ_h ⊓ τ_f
  4. 返回函数类型: (projected(x) -> infer(body))
\end{lstlisting}

\item \textbf{函数体内赋值}——收集\texttt{extendToBe}约束：
\begin{lstlisting}[language={}]
在函数体内: x = 42
如果x是参数:
  addExtend(x, Int)  // τ_e = τ_e ⊔ Int  (join, 从⊥向上扩展)
\end{lstlisting}

\item \textbf{使用上下文}——收集\texttt{hasToBe}约束：
\begin{lstlisting}[language={}]
在函数体内: let y: Number = x
如果x是参数:
  addDemand(x, Number)  // τ_h = τ_h ⊓ Number  (meet, 从⊤向下收紧)
\end{lstlisting}

\item \textbf{结构化操作}——收集\texttt{definedToBe}约束：
\begin{lstlisting}[language={}]
在函数体内: x + 1
如果x是参数:
  addStructure(x, Number)  // τ_f = τ_f ⊓ Number  (meet, 从⊤向下收紧)
\end{lstlisting}

\item \textbf{函数调用}——投影：
\begin{lstlisting}[language={}]
infer(f(arg)):
  1. 获取f的类型: τ_f = (τ_param -> τ_ret)
  2. 获取实参类型: τ_arg = infer(arg)
  3. 投影检查: τ_e ⊑ τ_arg ⊑ τ_d ⊑ τ_h ⊓ τ_f
  4. 成功: 参数实例化为τ_arg, 返回实例化的返回值类型
  5. 失败: 报告类型错误
\end{lstlisting}
\end{enumerate}

\paragraph{完整示例：}

\begin{lstlisting}[language={}]
let f = (x: Number) -> {
    x = 42;              // addExtend(x, Int):   τ_e = ⊥ ⊔ Int = Int
    let y: Number = x;   // addDemand(x, Number): τ_h = ⊤ ⊓ Number = Number
    return x + 1;        // addStructure(x, Number): τ_f = ⊤ ⊓ Number = Number
}
// 约束链: Int ⊑ Number ⊑ Number ⊓ Number = Number  ✓

f(100)   // Int ⊑ 100:Int ⊑ Number ⊑ Number  ✓  → 返回 Int
f(3.14)  // Int ⊑ 3.14:Float ⊑ Number ⊑ Number  ✓  → 返回 Float
f("hi")  // Int ⊑ "hi":String ⊑ Number  ✗  → 类型错误
\end{lstlisting}

\subsection{多模块推导：不动点算法}
\label{sec:fixpoint}

对于跨模块的程序，GCP使用不动点算法确保推导收敛：

\begin{algorithm}[h]
\caption{多模块不动点推导}
\label{alg:fixpoint}
\begin{algorithmic}[1]
\REQUIRE 模块集合 $\mathit{modules}$
\ENSURE 所有模块的推导类型
\STATE 初始化：为所有模块创建\texttt{LazySymbol}
\REPEAT
  \STATE $\mathit{changed} \leftarrow \mathbf{false}$
  \FOR{each $m \in \mathit{modules}$}
    \STATE $\mathit{newTypes} \leftarrow \texttt{inferModule}(m)$
    \IF{$\mathit{newTypes} \neq \mathit{oldTypes}$}
      \STATE $\mathit{changed} \leftarrow \mathbf{true}$;\quad $\mathit{oldTypes} \leftarrow \mathit{newTypes}$
    \ENDIF
  \ENDFOR
\UNTIL{$\neg\mathit{changed}$ \OR $\mathit{maxRounds}$ reached}
\RETURN 所有模块的推导类型
\end{algorithmic}
\end{algorithm}

\begin{theorem}[多模块全局收敛性]
\label{thm:global-conv}
对于$M$个模块、$N$个函数参数的程序，不动点算法在$O(M \cdot N \cdot |\mathbb{T}|)$步内终止。
\end{theorem}

\begin{theorem}[顺序无关性]
\label{thm:order}
推导结果与模块处理顺序无关。
\end{theorem}

\begin{proof}
不动点算法迭代直到没有变化，因此最终结果是所有约束的最小不动点，与处理顺序无关。\qed
\end{proof}

\paragraph{约束传播示例：}

def add(x, y):

\begin{enumerate}[leftmargin=2em]

\item \textbf{definedToBe}（从函数体操作符，meet$\sqcap$从$\top$向下）：
  \[\tau_f(\texttt{x}) = \top \sqcap \texttt{Number} = \texttt{Number}\]
\item \textbf{约束链}：$\bot \sqsubseteq \tau_{\mathit{arg}} \sqsubseteq \top \sqsubseteq \texttt{Number} \sqcap \texttt{Number} = \texttt{Number}$
\item \textbf{最终标注}：\texttt{def add(x: int | float, y: int | float) -> int | float}
\end{enumerate}

\subsubsection{联合推导与调用上下文}

GCP-Python的关键创新是\textbf{调用点驱动推导}：通过分析代表性调用点实际传递的参数类型来推导库函数参数的类型。

\textbf{输入：}\texttt{lib.py}（未标注函数的库文件）和\texttt{calls.py}（调用上下文文件）。\textbf{输出：}\texttt{lib\_annotated.py}（完全标注的库文件，可供mypyc编译）。

\subsection{Python转换器流水线}

Python的语法比Outline丰富得多。GCP-Python包含一个转换器流水线，在将Python AST馈送到GCP约束求解器之前规范化20种Python语法模式，包括：增强赋值（\texttt{x += 1}）、walrus操作符、下标、match/case、列表/字典推导、装饰器、with语句、异常处理、生成器、async/await等。

\subsection{函数单态化：FunctionSpecializer}

当函数在不同调用点以不同类型调用时，单一标注可能不够精确。GCP-Python使用\textbf{单态化}：为每个观察到的类型元组生成一个函数副本。

\begin{lstlisting}[language=Python]
# 原始函数
def add(x, y): return x + y

# 单态化后
def add_int_int(x: int, y: int) -> int: return x + y
def add_float_float(x: float, y: float) -> float: return x + y

# 调用点重写
z1 = add_int_int(10, 20)
z2 = add_float_float(1.5, 2.5)
\end{lstlisting}

\subsection{实现细节}

GCP-Python实现为约3000行Java代码，构建在GCP核心引擎之上。关键组件：\textbf{PythonConverter}（800行，Python AST转换为简化的GCP AST）、\textbf{PythonInferencer}（600行，Python特定的约束收集）、\textbf{FunctionSpecializer}（400行，单态化逻辑）、\textbf{AnnotationWriter}（300行，类型注入回Python源代码）、\textbf{GCP核心}（5000行，通用约束求解器）。

典型库文件（10--20个函数）分析耗时\textbf{66毫秒}，大型文件（100+函数）约300毫秒。

\paragraph{与Hindley-Milner的比较。}
H-M通过合一（unification）求解类型方程，在完全静态语言中可达到主类型（principal type）。GCP与H-M的核心区别在于：
(1)\textbf{约束来源}：H-M仅考虑函数体内的使用约束，GCP额外引入调用点约束（$\hasToBe$）；
(2)\textbf{动态扩展}：H-M不支持后期绑定，GCP通过$\extendToBe$的join操作自然处理多态赋值；
(3)\textbf{格语义}：GCP的格操作（$\sqcup/\sqcap$）比合一更灵活，可表达结构化子类型关系。

\paragraph{与TypeScript的比较。}
TypeScript采用双向类型流（bidirectional type flow）和结构化子类型，与GCP在结构化类型方面最为接近。主要区别：
(1)\textbf{顺序依赖}：TypeScript的推导依赖声明顺序，GCP通过不动点迭代实现顺序无关性；
(2)\textbf{调用点驱动}：TypeScript不从调用点反向推导参数类型，GCP的$\hasToBe$维度专门捕获此信息；
(3)\textbf{健全性}：TypeScript为工程实用性牺牲了部分健全性，GCP在有限假设下可证明健全。

\paragraph{与mypy的比较。}
mypy是Python的渐进类型检查器，与GCP-Python的目标最为接近。区别在于：
(1)\textbf{推导 vs 检查}：mypy主要验证已有标注，GCP主动推导标注；
(2)\textbf{调用上下文}：mypy不利用调用点信息推导参数类型，GCP的需求驱动推导正是基于此；
(3)\textbf{编译目标}：mypy的类型信息用于错误检测，GCP的类型信息直接驱动AOT编译优化。

\subsection{GCP的理论贡献}

\paragraph{格语义的统一框架。}
GCP将类型推导形式化为完全格上的不动点计算，统一了以往分散的约束处理方式。四个维度的初始值和单调方向由格的代数结构决定，而非启发式选择：
\begin{itemize}
  \item $\extendToBe$从$\bot$（$\NOTHING$）出发，通过$\sqcup$向上收敛，捕获"至少是什么"；
  \item $\hasToBe$和$\definedToBe$从$\top$（$\ANY$）出发，通过$\sqcap$向下收敛，捕获"至多是什么"；
  \item 最终约束链$\extendToBe \sqsubseteq \tau_{arg} \sqsubseteq \declaredToBe \sqsubseteq \hasToBe \sqcap \definedToBe$在格中形成一致的偏序关系。
\end{itemize}

\paragraph{OEM规则的基础性。}
OEM规则（$\NOTHING \sqsubseteq \tau$，$\tau \sqsubseteq \ANY$）不仅是语法糖，而是格的底元素和顶元素的直接体现。这使得GCP的类型系统在数学上是封闭的：任何类型推导序列都有明确的上下界，收敛性由Tarski不动点定理保证。

\paragraph{结构化子类型的格解释。}
GCP对结构化子类型的处理揭示了一个重要洞察：字段集合的并集（$\cup$）对应类型格中的meet（$\sqcap$），即更多字段意味着更具体的类型（更低的格元素）。这与直觉相符：$\{name, age\}$比$\{name\}$更具体，因为它满足更多约束。

\subsection{局限性}

\paragraph{高阶函数。}
当前GCP实现对高阶函数（函数作为参数传递）的处理有限。若参数本身是函数类型，$\hasToBe$维度需要捕获函数类型的调用签名，这需要扩展约束收集算法。

\paragraph{递归类型。}
对于相互递归的类型定义（如树结构），GCP的不动点迭代可能需要更多轮次才能收敛，最坏情况下复杂度可能退化。

\paragraph{动态特性。}
Python的某些动态特性（如\texttt{getattr}、\texttt{setattr}、元类）超出了GCP当前的分析能力。GCP-Python通过保守回退（将无法推导的参数标注为\texttt{Any}）处理这些情况。

\paragraph{调用上下文完整性。}
GCP假设分析时可获得代表性的调用上下文。若调用点不完整（如库函数），$\hasToBe$维度将保持$\top$（$\ANY$），退化为仅依赖$\definedToBe$的推导。

\subsection{未来工作}

\paragraph{跨模块推导。}
当前实现在单模块内进行不动点迭代。扩展到跨模块推导需要处理模块间的依赖关系，可借鉴过程间分析（interprocedural analysis）的技术。

\paragraph{增量推导。}
当代码发生局部修改时，重新运行完整推导代价较高。基于GCP的单调性，可设计增量更新算法，仅重新推导受影响的函数。

\paragraph{其他语言应用。}
GCP的格语义框架与语言无关。将GCP应用于JavaScript（动态类型）、Ruby或Lua等语言，可能产生类似的AOT编译收益。

\paragraph{与LLM的结合。}
大语言模型（LLM）可为GCP提供初始类型提示，缩短不动点迭代的收敛路径。反之，GCP推导的精确类型可作为LLM代码生成的约束，提高生成代码的类型安全性。

\section{结论}
%% ============================================================

本文提出了泛化约束投影（GCP）——一种基于完全格语义的四维类型推导框架，并在Python AOT编译场景中验证了其有效性。

\paragraph{理论贡献。}
GCP将函数参数的类型约束分解为四个正交维度（$\extendToBe$、$\declaredToBe$、$\hasToBe$、$\definedToBe$），每个维度在完全格上单调演化，通过Tarski不动点定理保证收敛。约束链$\extendToBe \sqsubseteq \tau_{arg} \sqsubseteq \declaredToBe \sqsubseteq \hasToBe \sqcap \definedToBe$在格中形成一致的偏序关系，统一了函数体内约束与调用点约束的处理。OEM规则（$\NOTHING \sqsubseteq \tau$，$\tau \sqsubseteq \ANY$）为类型系统提供了封闭的代数基础。

\paragraph{工程贡献。}
GCP-Python实现了需求驱动的Python AOT编译流水线：从Python源码出发，通过GCP推导精确类型标注，经mypyc编译为C扩展，无需程序员手工标注。在7个核心基准函数上，GCP-Python平均加速比达13.17$\times$（ARM64），达到手工标注性能的101.8\%，显著优于Numba JIT（7.45$\times$）和bare mypyc（1.30$\times$）。

\paragraph{核心洞察。}
GCP的成功揭示了一个普遍原则：\textbf{类型推导的本质是约束传播}，而约束传播的本质是格上的不动点计算。通过明确区分约束的来源（赋值/声明/调用点/结构操作）和方向（向上/向下），GCP在保持理论严谨性的同时，实现了对动态语言的实用类型推导。

\paragraph{展望。}
GCP框架具有良好的扩展性：可应用于其他动态语言、支持跨模块推导、与LLM结合提供类型提示。我们相信，基于格语义的约束推导将成为下一代编程语言工具链的重要基础。

\section{定理证明}

\subsection{定理3.1（单调性）的证明}

\begin{proof}
设$G = (G_e, G_d, G_h, G_f)$为Genericable对象，$\mathcal{L}$为完全格。

\textbf{addExtend的单调性：}
设$G_e \sqsubseteq G_e'$（即$G_e$在格中不高于$G_e'$）。
对任意类型$\tau$，$\text{addExtend}(G, \tau) = G[G_e \mapsto G_e \sqcup \tau]$。
由于$\sqcup$是格的最小上界操作，$G_e \sqcup \tau \sqsubseteq G_e' \sqcup \tau$（由$G_e \sqsubseteq G_e'$和$\sqcup$的单调性）。
因此$\text{addExtend}(G, \tau) \sqsubseteq \text{addExtend}(G', \tau)$，单调性成立。

\textbf{addDemand的单调性：}
设$G_h \sqsubseteq G_h'$。
$\text{addDemand}(G, \tau) = G[G_h \mapsto G_h \sqcap \tau]$。
由$\sqcap$的单调性，$G_h \sqcap \tau \sqsubseteq G_h' \sqcap \tau$。
因此$\text{addDemand}(G, \tau) \sqsubseteq \text{addDemand}(G', \tau)$，单调性成立。

\textbf{addStructure的单调性：}类似addDemand，证明略。

综上，$F$的每个分量均单调，故$F$整体单调。
\end{proof}

\subsection{定理3.2（收敛性）的证明}

\begin{proof}
由定理3.1，$F: \mathcal{L}^4 \to \mathcal{L}^4$是单调函数。

$\mathcal{L}$是完全格，故$\mathcal{L}^4$（积格）也是完全格。

由Tarski不动点定理~\cite{tarski1955}，完全格上的单调函数存在最小不动点$\mu F$，且可通过从底元素出发的迭代序列达到：
$$\bot \sqsubseteq F(\bot) \sqsubseteq F^2(\bot) \sqsubseteq \cdots \sqsubseteq \mu F$$

对于$\extendToBe$维度，初始值为$\bot = \NOTHING$，序列单调递增，在有限格中必然收敛。

对于$\hasToBe$和$\definedToBe$维度，初始值为$\top = \ANY$，序列单调递减（由meet操作的性质），在有限格中必然收敛。

由于类型格在实际程序中是有限的（类型数量有限），迭代在有限步内终止。
\end{proof}

\subsection{定理3.3（健全性）的证明}

\begin{proof}
设$\tau^* = \text{project}(G)$为GCP推导的类型，$\tau_{actual}$为运行时实际类型。

\textbf{前提假设：}
(A1) 调用点集合$\mathcal{C}$包含所有实际调用；
(A2) 函数体内的结构操作集合$\mathcal{S}$完整；
(A3) 类型格$\mathcal{L}$正确反映子类型关系。

\textbf{证明$\tau_{actual} \sqsubseteq \tau^*$：}

由(A1)，对所有实际调用$c \in \mathcal{C}$，$\tau_{arg}^c \sqsubseteq G_h = \hasToBe$（因为$G_h$是所有调用点类型的meet，是最大下界）。

由(A2)，对所有结构操作$s \in \mathcal{S}$，$\tau_{actual}$必须满足$s$的类型要求，故$\tau_{actual} \sqsubseteq G_f = \definedToBe$。

由约束链，$\tau^* = G_h \sqcap G_f$（忽略$G_d$和$G_e$的情况下），故$\tau_{actual} \sqsubseteq G_h$且$\tau_{actual} \sqsubseteq G_f$，由meet的定义，$\tau_{actual} \sqsubseteq G_h \sqcap G_f = \tau^*$。

因此，任何满足约束的实际类型都是$\tau^*$的子类型，健全性成立。
\end{proof}

\subsection{引理：hasToBe $\sqcap$ definedToBe的结构解释}

\begin{lemma}
在结构化子类型格中，若$\tau_1 = \{f_1, \ldots, f_m\}$，$\tau_2 = \{g_1, \ldots, g_n\}$，则：
$$\tau_1 \sqcap \tau_2 = \{f_1, \ldots, f_m\} \cup \{g_1, \ldots, g_n\}$$
即meet操作对应字段集合的并集。
\end{lemma}

\begin{proof}
在结构化子类型中，$\tau \sqsubseteq \sigma$当且仅当$\tau$的字段集合包含$\sigma$的字段集合（$\text{fields}(\sigma) \subseteq \text{fields}(\tau)$）。

$\tau_1 \sqcap \tau_2$是满足$\tau_1 \sqcap \tau_2 \sqsubseteq \tau_1$且$\tau_1 \sqcap \tau_2 \sqsubseteq \tau_2$的最大类型（最小上界的对偶）。

$\tau_1 \sqcap \tau_2 \sqsubseteq \tau_1$要求$\text{fields}(\tau_1) \subseteq \text{fields}(\tau_1 \sqcap \tau_2)$。
$\tau_1 \sqcap \tau_2 \sqsubseteq \tau_2$要求$\text{fields}(\tau_2) \subseteq \text{fields}(\tau_1 \sqcap \tau_2)$。

因此$\text{fields}(\tau_1 \sqcap \tau_2) \supseteq \text{fields}(\tau_1) \cup \text{fields}(\tau_2)$。

取最大（字段最少）满足条件的类型，即$\text{fields}(\tau_1 \sqcap \tau_2) = \text{fields}(\tau_1) \cup \text{fields}(\tau_2)$。
\end{proof}

此引理解释了为何$\hasToBe \sqcap \definedToBe$（meet）对应字段的并集：同时满足两个约束的类型必须拥有两者要求的所有字段，字段越多越具体，在格中位置越低（越接近$\bot$）。
